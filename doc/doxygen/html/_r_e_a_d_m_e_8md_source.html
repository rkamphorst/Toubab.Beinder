1 # Beinder: simple data binding
2 
3 From [Wikipedia](https://en.wikipedia.org/wiki/Data_binding):
4 
5 > Data binding is the process that establishes a connection between the application UI (User Interface) and Business
6 > logic. If the settings and notifications are correctly set, the data reflects changes when made. It can also mean that
7 > when the UI is changed, the underlying data will reflect that change.
8 
9 There are a lot of implementations of data binding, in C# and beyond. So why this library? I've written up a small
10 [justification](#justification) for myself below. It explains the main goals for Beinder:
11 
12 1. **Only do binding.**
13 2. **Be loosely coupled to the software that uses it.**
14 3. **Be adaptable through extension.**
15 4. **Be usable with minimal configuration.**

46 ### Pattern agnostic
47 
48 Although designed with the MVVM pattern in mind, the library does not know of views,
49 view models or business logic. It only knows of objects that have properties, events and
50 event handlers that need to be bound in some way.
51 
52 This results in the view and view model sides being treated exactly the same. The binder
53 only knows they are objects, not what function they have.
54 
55 And although in MVVM, you typically only bind two objects, it is possible to use the binder
56 to bind more than two. If, for some reason, you need to do this, just pass more objects to
57 `Binder.Bind()`.
58 
59 ### Name-based binding
60 
61 When you bind two (or more) objects, properties with the same name get bound to each other.
62 
63 * **The only way to bind two properties to each other is to give them the same name.**
64  There is no other (explicit) way to indicate that one property should be bound to
65  a specific property on another object.
66 
67 * **Bindings are two-way when possible.**
68  If two properties are bound to each other and one of them is read-only, obviously values
69  can only propagate from the read-only property to the write-enabled property. However,
70  when all bound properties are read-write, values can and will propagate from any of them
71  to any of them.
72 
73 * **No per-property data conversions can be specified**.
74  If a property could not be set because (for example) it has the wrong data type, this
75  will fail silently. However, if there are 'properties of properties' (a.k.a. *child properties*)
76  that match, those will be bound to each other instead
77  (see [Recursive binding](#recursive-binding)).
78 
79 Example:
80 
81  /* ViewModelClass and ViewClass both have a string property MyProperty,
82  * and they both implement INotifyPropertyChanged
83  */
84  var view = new ViewClass { MyProperty = "aaa" };
85  var viewModel = new ViewModelClass { MyProperty = "bbb" };
86 
87  this._bindings = binder.Bind(viewModel, view);
88 
89  // viewModel.MyProperty is now bound to view.MyProperty.
90  // view.MyProperty will have the value "bbb".
91 
92  view.MyProperty = "ccc"; // propagates "ccc" to viewModel.MyProperty
93  viewModel.MyProperty = "ddd"; // propagates "ddd" to view.MyProperty
94 
95 ### Recursive binding
96 
97 If values of properties that are bound have matching properties, those *child properties*
98 will be bound as well. This can be extended to child properties of child properties, and
99 so on.
100 
101 Example:
102 
103  /* ViewModelClass, ViewClass, ControlModelClass and ControlClass all
104  * implement interface INotifyPropertyChanged.
105  */
106  var view = new ViewClass() { MyControl = new ControlClass { Label = "aaa" } };
107  var viewModel = new ViewModelClass() { MyControl = new ControlModelClass { Label = "bbb" } };
108 
109  this._bindings = binder.Bind(viewModel, view);
110 
111  // viewModel.MyControl.Label is now bound to view.MyControl.Label.
112  // both will have the value "bbb".
113 
114  view.MyControl.Label = "ccc"; // propagates "ccc" to viewModel.MyControl.Label
115  viewModel.MyControl.Label = "ddd"; // propagates "ddd" to view.MyControl.Label
116 
117 ### Binding across object boundaries
118 
119 If a property's name is a concatenation of another property's name and the name of one of
120 it's child properties, the former property is bound to the child property. Again, this
121 can be extended to child properties of child properties, etc.
122 
123 Example:
124 
125  /* ViewModelClass, ViewClass, and ControlClass all implement interface
126  * INotifyPropertyChanged.
127  */
128  var view = new ViewClass() { MyControl = new ControlClass { Label = "aaa" } };
129  var viewModel = new ViewModelClass() { MyControlLabel = "bbb" };
130 
131  this._bindings = binder.Bind(viewModel, view);
132 
133  // viewModel.MyControlLabel is now bound to view.MyControl.Label.
134  // both will have the value "bbb".
135 
136  view.MyControl.Label = "ccc"; // propagates "ccc" to viewModel.MyControlLabel
137  viewModel.MyControlLabel = "ddd"; // propagates "ddd" to view.MyControl.Label

147 ### Value change propagation
148 
149 When `Binder.Bind()` is called, properties are bound, and then values are propagated.
150 The values are propagated from properties on the first parameter to properties
151 on the other parameters. Recall that in in the previous examples, the bound
152 properties would initially be set to the value of the `viewModel`'s property: this
153 is because `viewModel` was always the first parameter to `Bind()`.
154 
155 After a property has been bound, and its value changes, its value can be propagated to the
156 other properties it is bound to. However, for this to happen, an event is needed that is
157 raised whenever the property changes. This needs to be implemented in the class/object the
158 property comes from. There are two standard ways in order to do this:
159 
160 1. Implement `INotifyPropertyChanged` on the objects you want to bind.
161  See also
162  [NotifyPropertyChangedScanner](Toubab.Beinder/PropertyScanners/NotifyPropertyChangedScanner.cs).
163 2. Implement an event (handler type `EventHandler`) for each property you wan to bind,
164  with the same name, but postfixed with "Changed". Raise it whenever the property is
165  set with a new value.
166  See also
167  [ReflectionScanner](Toubab.Beinder/PropertyScanners/ReflectionScanner.cs).
168 
169 ### Dynamic rebinding
170 
171 Whenever a property's value changes, and this new value is propagated to other properties,
172 what happens to the property's *child properties*? They get rebound.
173 
174 Consider the following example.
175 
176  /* ViewModelClass, ViewClass, ControlModelClass, ControlClass and TextControlClass all
177  * implement interface INotifyPropertyChanged.
178  */
179  var view = new ViewClass() { MyControl = new ControlClass { Label = "aaa" } };
180  var viewModel = new ViewModelClass() { MyControl = new ControlModelClass { Label = "bbb", Text = "ccc" };
181 
182  this._bindings = binder.Bind(viewModel, view);
183 
184  // viewModel.MyControl.Label is now bound to view.MyControl.Label.
185  // Both will have the value "bbb".
186  // viewModel.MyControl.Text is not bound to anything and will have value null.
187 
188  view.MyControl = new TextControlClass { Text = "ddd" };
189 
190  // viewModel.MyControl.Text gets *rebound* to view.MyControl.Text.
191  // Both will now have value "ddd".
192  // viewModel.MyControl.Label will still have value "bbb".

206 ## Justification
207 
208 So why roll my own binding library?
209 
210 (Data) binding is implememented as part of many frameworks, for many languages and for many programming environments (see
211 the mentioned wikipedia page for more examples). Examples for C# include a myriad of UI frameworks that Microsoft has
212 made over the years (Windows Forms, WPF, ASP.NET, ...), and the last couple of years the various MVVM frameworks that
213 have sprung up to support cross-platform app development in C# (examples are
214 [MvvmCross](https://github.com/MvvmCross/MvvmCross), [MVVM Light Toolkit](https://mvvmlight.codeplex.com/),
215 [Xamarin Forms](https://xamarin.com/forms)).
216 
217 Most existing implementations, IMHO, have the following drawbacks:
218 
219 1. They are *part of* a larger framework, and cannot easily be used outside that framework.
220  Especially MvvmCross has a splendid binding implementation; however, choosing its binding framework
221  without pulling in the rest into your project is difficult (I did succeed, though).
222 2. They are complex and hard to configure. This is mostly solved by having great IDE support and
223  great documentation. However, if you do not possess the right IDE and/or have a specific problem that
224  is not covered by IDE or docs, or there is a small bug that needs to be solved, you need to dive
225  into the framework's code and learn all of its intricate details.
226 3. They give too much freedom in the wrong places. This results in complexity where it is not needed,
227  code that is difficult to read, and View-ViewModel bindings that are hard to track.
228 
229 Therefore, the goals of Beinder are to:
230 
231 1. **Only do binding.**
232  ...but do it as well as any other framework, or better.
233 2. **Be loosely coupled to the software that uses it.**
234  The code/configuration footprint related to the use of this library should be as small as possible and in a small number of places.
235 3. **Be adaptable through extension.**
236  Any aspect of the library that could need adaptation, e.g. to a specific platform or toolkit, should be adaptable through extension.
237 4. **Be usable with minimal configuration.**
238  Have a sensible default configuration that handles almost all of the cases, and use *adaptation through extension* for the rest.
239 
240 However, the most important reason to make this library is... well because it's fun :-)
